1) Put the framework in replit.md so Agent always follows it

Replit has a built-in mechanism for this: replit.md in the project root. Agent automatically reads it and uses it as context for future work.

What to include in replit.md (keep it short, but strict):

A. Non‑negotiables

All flows must be launched via startProcess(processId, { returnTo })

All completions must go through finishProcess() (primary = returnTo, secondary = view page)

All navigation must use safeNavigate() / <GuardedLink /> (no direct router calls)

Any flow with data entry must register UnsavedChangesGuard

Top tabs always visible (Process runs inside AppShell)

B. “How to add a new Process”

A checklist that Agent must follow whenever it adds/modifies a process:

Create process component using the template wrapper

Register it in processRegistry.ts (single source of truth)

Add routes only via the registry

Use useProcessReturn() to return to origin

Implement dirty guard if anything can be unsaved

Add at least 1 smoke test path: “start → complete → returns to origin”

C. File placement rules

Example:

client/src/processes/<processId>/...

client/src/processes/registry.ts

client/src/navigation/safeNavigate.ts

client/src/guards/unsavedChangesGuard.tsx

Why this helps: instead of each feature inventing structure, every feature lands in the same place.

Important: Keep replit.md focused. Replit warns that extremely large files may not be fully processed, and recommends being specific + using examples.

2) Add a Replit “Skill” so the framework persists across chats

Replit’s Agent Skills are designed specifically to prevent “context drift” between sessions: skills preserve what you’ve learned together so the agent applies it consistently later.

Do this once:

Create a skill in /.agents/skills/process-framework.md that contains:

your process framework rules

“dos/don’ts”

a short code snippet for the standard Process wrapper

a short code snippet for safe navigation + guard pattern

Then after you fix any tricky bug once, you can literally tell Replit:

“Turn this into a skill so you remember it.”

That’s exactly what Skills are for.

3) Make it hard to break the framework with code guardrails

This is the most underrated step. You can encode the rules so “wrong” patterns fail loudly.

A. Centralize and “ban” direct navigation

Create only one way to navigate:

safeNavigate(path)

<GuardedLink to="/...">

Then in ESLint, restrict direct router usage:

Use no-restricted-imports to block importing useLocation / raw navigation helpers outside client/src/navigation/*

This forces new code to go through your guard layer

B. Force all processes to use a wrapper

Create a ProcessLayout component that requires:

processId

returnTo

onComplete()

viewPath (optional)

If someone makes a “process page” without it, it won’t match your conventions, and it will be obvious in code review (or you can enforce via lint rule / folder conventions).

C. Single source of truth registry

Create processRegistry.ts like:

id

title

component

defaultViewPath

requiresDirtyGuard: boolean

Then:

routes are generated from registry

the top-level process router renders by lookup

This stops the “random new routes” problem and makes the app truly building-block based.

4) Add automated “quality gates” that every Agent change must pass

You want a single command that tells you: “the codebase is still clean.”

A. Add a verify script

Example behavior:

Typecheck

Lint

Unit tests (even minimal)

(Optional) healthcheck scripts

Then in every Replit prompt, require:

“Run npm run verify and fix failures.”

B. Add a tiny test for the framework itself

Even 3–5 tests dramatically reduce regressions:

returnTo sanitization (no open redirects)

safeNavigate blocks when dirty

finishProcess goes to returnTo by default

These tests protect your core UX promise.

5) Use a repeatable prompting pattern (so Agent does consistent work)

Replit’s own prompting guidance: plan first, build incrementally, be specific, show examples, provide relevant files.

So instead of “add a new tab,” you use a predictable template prompt every time:

Your “always use this” prompt skeleton

Copy/paste this as a header for almost every new feature request:

Before coding:
1) Read replit.md and the process framework skill.
2) Describe the plan in 5–10 bullets and list files you will touch.

While coding:
- All navigation must go through safeNavigate/GuardedLink.
- All flows must use startProcess/finishProcess + returnTo default.
- If there are unsaved edits, register UnsavedChangesGuard.

After coding:
- Run npm run verify and fix issues.
- Provide: files changed, what you added, and smoke steps.

This prevents the “Agent made it work but didn’t integrate it properly” outcome.

6) What I’d do next in your repo (concrete setup checklist)

If you haven’t already, do these once:

Create/update replit.md with the framework rules + examples.

Create /.agents/skills/process-framework.md to persist the patterns across sessions.

Add:

processRegistry.ts

startProcess() / finishProcess()

safeNavigate() + <GuardedLink />

UnsavedChangesProvider + confirm modal

Add ESLint restrictions so direct navigation gets blocked.

Add npm run verify and require it in every Agent prompt.

Add 3 minimal tests for the navigation/guard framework.

The “why” in one sentence

If you only rely on prompts, you’ll get drift. If you bake the framework into replit.md + skills + lint + tests, every new page/process is forced into the same rails automatically.

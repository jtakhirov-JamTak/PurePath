1) Unify entitlement logic across docs

Right now:

Payment Flow’s server-side entitlement check includes status === 'completed' (good). 

PAYMENT-FLOW

Routes.md’s requireEntitlement (server-side example) does not check status. 

ROUTES

Routes.md’s PaidRoute (frontend UX gate) does not check status either. 

ROUTES

Payment Flow’s /api/user/entitlements example also doesn’t check status. 

PAYMENT-FLOW

Why this matters: as soon as you add refunds/chargebacks or “pending” purchases, the UI can show access while the API denies it (or vice versa).

What to change (simple rule):

Server requireEntitlement: always check status === 'completed'.

Frontend “paid route” check: also check status === 'completed' so the UI matches server truth.

/api/user/entitlements: include status check too.

(Your Security doc already recommends “fail closed” patterns; this aligns perfectly.) 

SECURITY

2) Align the checkout success/cancel routes with Paywall UX

Paywall UX says:

after payment: redirect to /checkout/success?session_id=... then return to returnUrl

cancel goes to /checkout/cancel 

PAYWALL-UX

Payment Flow currently uses:

success_url: .../dashboard?success=true

cancel_url: .../checkout?canceled=true 

PAYMENT-FLOW

Pick one and make all docs match.
Given your Paywall UX is more detailed and conversion-oriented, I’d adjust Payment Flow to match Paywall UX (use /checkout/success + /checkout/cancel) and then update Routes.md route map accordingly.

Extra reliability tip: On /checkout/success, it’s worth polling /api/purchases or /api/user/entitlements for a few seconds before redirecting to returnUrl, because Stripe redirect can happen before your webhook processing completes. You already have a “Refresh Access” concept; this just makes success smoother.

3) Add /api/checkout to Routes.md and clarify auth expectation

Routes.md currently lists /checkout but doesn’t list /api/checkout (even though Payment Flow uses it).

To prevent Agent confusion, I’d update Routes.md route map to include:

POST /api/checkout → isAuthenticated required

/checkout UI route can remain public (it just prompts login or calls API after login)

This also lines up with the Milestones doc’s “Payments integration” build step.

4) Enforce the “one journal per user per date” rule in the database

Data Model says “one entry per user per date (upsert pattern).” 

DATA-MODEL

But it doesn’t explicitly define the unique constraint that makes that guarantee real.

Recommendation: add a unique constraint/index:

UNIQUE (user_id, date)

Without it, an Agent may implement upsert logic but still allow duplicates in edge cases.

5) Decide how strict course_type should be (and encode it)

Your app logic depends heavily on course_type values being correct (course1, course2, bundle). It’s worth enforcing at the DB layer.

Right now Data Model describes it as free-form varchar. 

DATA-MODEL

Recommendation: either:

use a Postgres enum (course_type_enum) and map it in Drizzle, or

use a check constraint.

This prevents silent typos like "Course1" breaking access checks.
